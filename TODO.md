## 开发协议（每轮必须遵守）

1. **按序执行**：从最靠前未完成的 Phase 开始，不跳跃、不并行。
2. **一次一勾**：每轮只推进并勾选**一个**子任务；完成后再进下一个。
3. **每轮四要素**：动手前先用下面模板写清楚四点，再写代码。
4. **每模块测四态**：每个 Phase 收尾必须验证 ①主流程 ②加载中 ③数据为空 ④接口报错（含鉴权 403）四种状态。
5. **测试通过即推送**：四态通过后立即 `git commit` 并 `git push`，再进入下一个 Phase。
6. **守约束**：不碰功能黑名单（PRD §5），不破坏承重墙（ARCHITECTURE §8 B1–B11）。

### 每轮四要素模板（开工前填写）
```
▶ 当前任务：Phase X - 子任务名
① 修改目标：本轮要达成的唯一结果
② 允许修改范围：明确到目录/文件，范围外一律不动
③ 不允许破坏的逻辑：列出涉及的 B编号（ARCHITECTURE §8）
④ 验收标准：可观察的通过条件 + 四态测试点 + 提交信息
```

### 四态测试清单（每 Phase 收尾逐项核对）
- [ ] **主流程**：核心接口/页面按预期正常工作（对齐 PRD 黄金路径相关环节）
- [ ] **加载中**：异步请求期间有 loading（骨架屏/spinner，DESIGN §6.1）
- [ ] **数据为空**：无数据时显示空态文案（DESIGN §6.2）
- [ ] **接口报错**：后端 5xx/4xx 与越权 403 时有错误态（DESIGN §6.3），且全局异常统一包装为 `Result`

### 提交规范
- 信息格式：`feat(模块): 简述` / `fix(模块): 简述` / `chore: 简述`
- 流程：`git add -A` → `git commit -m "..."` → `git push`（如未配置远程，先 `git remote add origin <url>`）
