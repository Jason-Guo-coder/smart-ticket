# SmartTicket 智工单 · 产品需求文档（PRD）

> 版本：v1.1（MVP）｜更新日期：2026-06-21｜文档负责人：项目作者
>
> 一句话定义：**面向企业内部的智能工单 / 报修平台**——报修人一句话描述问题，系统借助 AI 自动分类、定优先级并给出派单建议，工程师按标准流程处理，管理员看板调度，全链路可追溯、可统计。

本文件是产品层的唯一事实来源（Single Source of Truth）。设计实现见 [`DESIGN.md`](./DESIGN.md)，技术实现见 [`ARCHITECTURE.md`](./ARCHITECTURE.md)。三份文档如有冲突，**业务规则以本文件为准，技术约束以 ARCHITECTURE.md 为准，视觉规范以 DESIGN.md 为准**。

## 1. 产品定位

### 1.1 是什么
SmartTicket 是一个**轻量级、可私有化部署的企业内部工单管理系统**，覆盖「提交 → 派单 → 处理 → 验收 → 评价」的完整工单生命周期。典型适用场景：

- **IT 运维报障**：电脑、网络、账号、软件故障
- **后勤 / 物业报修**：水电、设备、门禁、空调
- **设备维护工单**：生产 / 实验 / 办公设备

### 1.2 解决什么问题
把企业里**靠微信群、电话、Excel 登记的混乱报修流程**，收敛成一个**有状态、可追踪、可考核**的标准化系统：每一次报修都有唯一工单号、明确状态、完整时间线和最终沉淀。

### 1.3 差异化定位（防"烂大街"）
- **不做**电商 / 外卖 / 博客这类被做烂的方向，选择真实存在但少有人做的"企业内部工单"刚需场景。
- **AI 只做一件小而准的事**：把报修人的自然语言描述结构化为「类别 + 优先级 + 派单建议 + 相似历史工单」，降低派单的人工判断成本。AI 是**点睛模块，不喧宾夺主**。
- **工程深度优先**：RBAC 动态权限、工单状态机、并发抢单 / 派单、限流、幂等、审计日志、事务一致性等后端硬功夫才是系统主体。

### 1.4 目标用户与价值

| 角色 | 谁 | 核心诉求 | 系统提供的价值 |
|---|---|---|---|
| 报修人（员工） | 任意员工 | 快速提交、随时查进度、对结果评价 | 统一入口、状态透明、一句话提交 |
| 工程师（处理人） | IT / 后勤 / 设备工程师 | 看到分给自己的工单、按流程处理、记录方案 | 待办聚焦、优先级排序、知识沉淀 |
| 管理员（调度 / 主管） | 运维主管 | 全局看板、派单调度、SLA 监控、绩效统计 | AI 辅助派单、超时预警、绩效排行 |

## 2. 用户痛点

| # | 痛点 | 现状（无系统） | SmartTicket 的解法 |
|---|---|---|---|
| P1 | 报修渠道乱 | 微信群 / 电话 / 口头，易遗漏 | 统一入口，工单全部留痕 |
| P2 | 派单靠人盯 | 主管手动判断派给谁，慢且易积压 | AI 自动给类别 / 优先级 / 派单建议，一键采纳 |
| P3 | 进度不透明 | 报修人不知道修没修、谁在修 | 状态实时可查 + 处理时间线 |
| P4 | 处理无标准 | 修完即结束，经验不沉淀 | 强制记录解决方案，沉淀为可复用知识 |
| P5 | 无法考核 | 谁干得多干得好说不清 | 工程师绩效统计 + 排行榜 + 满意度评价 |
| P6 | 重复问题反复修 | 同样的故障每次从零排查 | AI 推荐相似历史工单及其解决方案 |
| P7 | 响应超时无人知 | 工单躺着没人管 | SLA 超时自动标红 + 定时扫描预警 |

## 3. 核心用户旅程

> 旅程描述"用户在系统里如何走完一件事"，对应原型中的具体页面。页面清单见 §6。

### 3.1 黄金路径（端到端演示主线）
这是验收与 Demo 的主线，必须完整跑通：

```
报修人提交「我的电脑连不上公司 WiFi」
  → AI 0.8s 返回：类别=网络 / 优先级=高 / 建议派给张伟 / 相似工单 3 条
  → 管理员在派单页一键「采纳 AI 建议派单」
  → 工程师张伟接单 → 处理中 → 填写解决方案 → 提交完成（待验收）
  → 报修人验收通过 → 5 星评价
  → 管理员看板「已完成 +1」，张伟绩效排行上升
```

### 3.2 报修人旅程
1. **登录** → 进入「我的工单」。
2. **提交工单**（提交工单页）：填写标题 + 描述 +（可选）图片 → 提交后右侧实时展示 AI 分析结果（类别 / 优先级 / 建议工程师 / 相似历史工单）。
3. **跟进进度**（我的工单列表 / 工单详情）：按状态筛选、查看处理时间线、SLA 倒计时、解决方案。
4. **验收与评价**（评价弹窗）：工单完成后对服务打分（星级）+ 选标签 + 补充评论。

### 3.3 工程师旅程
1. **登录** → 进入「待办工单」（按优先级排序，顶部显示待接单 / 处理中 / 今日完成 / 平均时长）。
2. **接单与处理**（工单处理页）：接单 → 状态流转为处理中 → 填写解决方案 → 上传处理截图（可选）→ 提交完成（转待验收）；必要时**驳回**或**转派**。
3. **查看绩效**（我的绩效页）：本月处理量、平均时长、满意度、准时率、团队排名、满意度分布。

### 3.4 管理员旅程
1. **登录** → 进入「工单看板」：KPI（总数 / 待派单 / 处理中 / 今日完成 / SLA 超时）、状态分布、近 7 天趋势、SLA 超时预警、类别分布。
2. **派单调度**（派单管理页）：对待派单工单，选择「采纳 AI 建议 / 手动指派 / 自动分配」；右侧实时参考工程师负载。
3. **团队管理**（工程师管理页）：查看工程师在线状态、技能组、当前负载、本月完成、满意度、绩效排行榜。
4. **账号与权限**（用户与角色页）：增删用户、分配角色（RBAC）。
5. **追溯审计**（审计日志页）：按操作类型 / 时间筛选，查询关键操作留痕。

## 4. MVP 功能列表（In Scope）

> 原则：**主体业务做完整、做扎实，AI 做一个小而可演示的模块**。每个功能都标注了归属模块（见 ARCHITECTURE.md §2）。

| 编号 | 功能 | 说明 | 模块 |
|---|---|---|---|
| F1 | 账号体系 | 登录 / 登出；三类角色（报修人 / 工程师 / 管理员） | auth |
| F2 | RBAC 权限 | 角色—权限—接口映射，接口级动态鉴权 | user |
| F3 | 工单提交 | 标题 + 描述 +（可选）图片，自动生成工单号 | ticket |
| F4 | **AI 辅助分析** | 提交时返回「类别 + 优先级 + 派单建议 + 相似历史工单」，失败可降级 | ai |
| F5 | 派单 | 管理员手动指派 / 一键采纳 AI 建议 / 自动分配（按类别 + 负载） | dispatch |
| F6 | 工单处理 | 工程师接单、状态流转、填写解决方案、驳回 / 转派 | ticket |
| F7 | 工单状态机 | 待派单 → 处理中 → 待验收 → 已完成 → 已评价（含驳回 / 转派回退） | ticket |
| F8 | 工单查询 | 多条件分页查询 + 详情时间线 | ticket |
| F9 | 评价 | 报修人对完成工单打分 + 标签 + 评论 | evaluation |
| F10 | 看板与统计 | 工单量、状态分布、趋势、工程师绩效排行 | stats |
| F11 | SLA 监控 | 超时工单自动标记 + 定时扫描预警 | sla |
| F12 | 操作审计日志 | 关键操作全程留痕，可追溯查询 | common + 各模块 |

### 4.1 工单状态机（业务核心）

```
[待派单] --派单--> [处理中] --提交完成--> [待验收] --验收通过--> [已完成] --评价--> [已评价]
   ▲                  │                        │
   │               --转派-->[待派单]         --验收驳回-->[处理中]
   └──────────(管理员重新分配)
```

- 合法流转**仅限上述路径**，任何越级 / 逆向流转必须被服务层拒绝（见 ARCHITECTURE.md 禁止破坏的逻辑）。
- 每次流转都要写 `ticket_log`（时间线 + 审计双用途）。

### 4.2 AI 模块行为约定（产品视角）
- **入参**：工单标题 + 描述。
- **出参**：`{ category, priority, suggestedEngineer, similarTickets[] }`。
- **时效**：同步调用，超时阈值 3s；解析失败重试 1 次。
- **降级**：任一环节失败 → 工单照常进入「待派单」，仅标记"无 AI 建议"，留待人工。**AI 失败绝不能阻断主流程。**
- **相似工单（MVP）**：用「类别 + 关键词 LIKE」实现；向量检索为进阶项，不进 MVP。

## 5. 功能黑名单（Out of Scope，明确砍掉）

> 这些是**本期坚决不做**的事项，用于守住范围、防止范围蔓延（对应风险 R4）。

| 黑名单项 | 原因 / 替代方案 |
|---|---|
| ✂️ 多租户 / 多组织隔离 | MVP 仅单组织 |
| ✂️ 消息推送（短信 / 邮件 / IM） | 用站内状态轮询代替；**约束：不引入 MQ** |
| ✂️ 工单流程自定义编排 | 流程固定即可，不做流程引擎 |
| ✂️ 移动端原生 App | Web 响应式即可 |
| ✂️ 复杂 BI 报表 / 数据大屏 | 看板满足基础统计即可 |
| ✂️ 附件对象存储 | MVP 用本地存储或仅存 URL |
| ✂️ 向量检索相似工单 | 用「类别 + 关键词 LIKE」代替；列为进阶加分项 |
| ✂️ 微服务 / Spring Cloud | 单体应用，禁止拆分 |

## 7. 验收标准（Definition of Done）

### 7.1 主线验收（必须）
一条工单能**端到端跑通** §3.1 黄金路径：报修人提交 → AI 给出建议 → 管理员派单 → 工程师处理并记录方案 → 报修人评价 → 管理员看板看到统计变化。

### 7.2 分项验收

| 维度 | 验收点 |
|---|---|
| 权限 | 每个核心接口都有 RBAC 校验；越权访问返回 403 |
| 状态机 | 非法流转被拒绝；每次流转写入 ticket_log |
| 并发 | 同一待派单工单并发派单只成功一次（乐观锁 + 分布式锁） |
| AI 容错 | AI 超时 / 返回非法 JSON 时自动降级，主流程不受影响 |
| 审计 | 登录、派单、状态流转、评价、用户 / 权限变更均有日志，可按条件查询 |
| SLA | 定时任务能扫描并标记超时工单，看板高亮预警 |
| 幂等 | 重复提交工单 / 重复评价被拦截 |
| 端到端 | 三类角色各自旅程在前端可完整操作演示 |

### 7.3 非功能性底线
- AI 故障时系统可**人工兜底**，不影响工单流转。
- 核心写操作（派单、状态变更、评价）均在**事务**内完成「业务 + 写日志」。
- Demo 前需预跑黄金路径，确保可现场演示。

## 8. 主要风险

| # | 风险 | 影响 | 应对 |
|---|---|---|---|
| R1 | AI 喧宾夺主 | 偏离"后端为主体"定位 | 严格限定 AI 仅 1 个模块；优先打磨 RBAC / 并发 / 事务 |
| R2 | AI 接口不稳定 / 有成本 | 演示翻车 | 同步调用设超时 + 降级兜底；结果缓存；Demo 前预跑 |
| R3 | AI 结构化输出不可靠 | JSON 解析失败 | Prompt 强约束 + 校验 + 重试 1 次 + 失败降级 |
| R4 | 范围蔓延（想加 MQ / 微服务） | 做不完 | 严守 §5 功能黑名单 |
| R5 | 并发派单重复指派 | 数据不一致 | 乐观锁 + Redis 分布式锁 + 事务 |
| R6 | 前端拖累进度 | 时间花错地方 | 用 Element Plus 现成组件，前端"能演示"即可 |
| R7 | 被视为普通 CRUD | 缺亮点 | 准备亮点话术：限流 / 幂等 / 状态机 / RBAC / AI 工程化 |
| R8 | 相似检索做太重 | 复杂度爆炸 | MVP 用类别 + 关键词 LIKE，向量检索列为进阶 |
