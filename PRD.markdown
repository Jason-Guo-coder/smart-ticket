# SmartTicket 智工单 · 产品需求文档（PRD）

> 版本：v1.0（MVP）  |  更新日期：2026-06-20  |  文档负责人：项目作者
>
> 一句话定义：**面向企业内部的智能工单/报修平台**——报修人一句话描述问题，系统借助 AI 自动分类、定优先级并给出派单建议，工程师按流程处理，管理员看板调度，全链路可追溯、可统计。

---

## 1. 产品定位

### 1.1 是什么
SmartTicket 是一个**轻量级、可私有化部署的企业内部工单管理系统**。覆盖"提交 → 派单 → 处理 → 完成 → 评价"的完整生命周期，适用于：

- IT 运维报障（电脑/网络/账号/软件问题）
- 后勤/物业报修（水电、设备、门禁、空调）
- 设备维护工单（生产/实验/办公设备）

### 1.2 解决什么问题
把企业里**靠微信群、电话、Excel 登记的混乱报修流程**，收敛成一个**有状态、可追踪、可考核**的标准化系统。

### 1.3 与同类产品的差异点（防"烂大街"）
- **不做** 电商/外卖/博客这类被做烂的方向，选择真实存在但少有人做的"内部工单"刚需场景。
- **AI 只做一件小而准的事**：把报修人的自然语言描述结构化（类别 + 优先级 + 派单建议 + 相似历史工单），降低派单人工成本——AI 是点睛模块，不喧宾夺主。
- **工程深度优先**：RBAC 动态权限、状态机、并发抢单、限流、审计日志等后端硬功夫是主体。

### 1.4 目标用户
| 角色 | 典型诉求 |
|---|---|
| 报修人（员工） | 快速提交、随时查进度、对结果评价 |
| 工程师（处理人） | 看到分给自己的工单、按流程处理、记录解决方案 |
| 管理员（调度/主管） | 全局看板、派单调度、SLA 监控、绩效统计 |

## 2. 用户痛点

| # | 痛点 | 现状（无系统） | SmartTicket 的解法 |
|---|---|---|---|
| P1 | 报修渠道乱 | 微信群/电话/口头，易遗漏 | 统一入口，工单全部留痕 |
| P2 | 派单靠人盯 | 主管手动判断该派给谁，慢且易积压 | AI 自动给类别/优先级/派单建议，一键采纳 |
| P3 | 进度不透明 | 报修人不知道修没修、谁在修 | 状态实时可查 + 时间线 |
| P4 | 处理无标准 | 修完即结束，没沉淀 | 强制记录解决方案，沉淀为可复用知识 |
| P5 | 无法考核 | 谁干得多干得好说不清 | 工程师绩效统计 + 排行榜 + 满意度评价 |
| P6 | 重复问题反复修 | 同样的故障每次从零排查 | AI 推荐相似历史工单及其解决方案 |
| P7 | 响应超时无人知 | 工单躺着没人管 | SLA 超时自动标红/提醒 |

## 3. MVP 范围

> 原则：**主体业务做完整、做扎实，AI 做一个小而可演示的模块**。砍掉一切非核心的"锦上添花"。

### 3.1 MVP 必做（In Scope）
- [x] 账号体系：登录/登出、三类角色（报修人/工程师/管理员）
- [x] 权限：基于 RBAC 的接口级权限控制
- [x] 工单提交：标题 + 描述 +（可选）图片，自动生成工单号
- [x] **AI 辅助**：提交时调用 AI 返回「类别 + 优先级 + 派单建议 + 相似历史工单」
- [x] 派单：管理员手动指派 + 一键采纳 AI 建议 + 自动分配（按类别/负载）
- [x] 工单处理：工程师接单、流转状态、填写解决方案
- [x] 状态机：待派单 → 处理中 → 待验收 → 已完成 → 已评价（含「驳回/转派」）
- [x] 工单查询：多条件分页查询、详情时间线
- [x] 评价：报修人对完成工单打分 + 评论
- [x] 看板与统计：工单量、各状态分布、工程师绩效排行
- [x] SLA：超时工单自动标记 + 定时扫描
- [x] 操作审计日志

### 3.2 MVP 不做（Out of Scope，明确砍掉）
- ✂️ 多租户 / 多组织隔离
- ✂️ 消息推送（短信/邮件/IM）——用站内状态轮询代替（约束里也不用 MQ）
- ✂️ 工单流程自定义编排（流程固定即可）
- ✂️ 移动端 App（Web 响应式即可）
- ✂️ 复杂报表 BI、数据大屏
- ✂️ 附件走对象存储（MVP 用本地存储或仅存 URL）

### 3.3 验收标准（Definition of Done）
一条工单可以**端到端跑通**：报修人提交 → AI 给出建议 → 管理员派单 → 工程师处理并记录方案 → 报修人评价 → 管理员在看板看到统计变化。且核心接口有权限校验、有审计日志、AI 故障时能人工兜底。

## 5. 核心流程

### 5.1 工单状态机
```
[待派单] --派单--> [处理中] --提交完成--> [待验收] --验收通过--> [已完成] --评价--> [已评价]
   ^                  |                        |
   |               --转派-->[待派单]        --验收驳回-->[处理中]
   +----------------(管理员重新分配)
```

### 5.2 主流程：提交 → AI 辅助 → 派单
```
报修人提交工单
   ↓
后端落库（状态=待派单，生成工单号）
   ↓
异步/同步调用 AI 模块：
   入参：标题+描述
   出参：{ 类别, 优先级, 派单建议工程师, 相似历史工单[] }
   ↓（AI 超时/失败 → 降级：跳过，留待人工）
管理员在派单页：
   - 采纳 AI 建议一键派单  /  手动指派  /  自动分配
   ↓
分配成功（乐观锁防并发重复派单），状态=处理中
```

### 5.3 处理与验收流程
```
工程师接单 → 处理中 → 填写解决方案 → 提交完成（待验收）
   ↓
管理员/报修人验收：
   通过 → 已完成 → 报修人评价 → 已评价
   驳回 → 回到处理中（附驳回原因）
```

### 5.4 AI 模块时序（重点说明工程细节）
```
提交工单
  → 拼装 Prompt（要求结构化 JSON 输出：category/priority/suggestedEngineer/similarTickets）
  → 调用大模型 API（设超时，如 3s）
  → 解析并校验 JSON（失败则重试 1 次）
  → 命中结果写入 Redis 缓存（相同描述短期内复用，控成本）
  → 任一环节失败 → 降级为「无 AI 建议」，工单照常进入待派单
```
> 相似工单检索 MVP 简化方案：先用「类别 + 关键词 LIKE」实现；进阶可加向量检索（作为可选加分项，不进 MVP）。

## 6. 技术方案

### 6.1 技术栈（限定）
| 层 | 选型 |
|---|---|
| 框架 | Spring Boot |
| ORM | MyBatis-Plus |
| 数据库 | MySQL |
| 缓存 | Redis |
| 认证 | JWT |
| 授权 | Spring Security（RBAC） |
| AI | 大模型 HTTP API（结构化输出） |
| 前端 | Vue3 + Element Plus（演示用，可替换） |

> 约束：**不使用 RabbitMQ、不使用 Spring Cloud**，整体为单体应用。

### 6.2 模块划分
```
smart-ticket
├── auth        登录/JWT 签发与校验/登出（token 黑名单）
├── user        用户、角色、权限（RBAC）
├── ticket      工单核心：提交/查询/状态流转
├── dispatch    派单：手动/自动/采纳AI建议
├── ai          AI 辅助分类与相似检索（含降级）
├── evaluation  评价
├── stats       看板与绩效统计
├── sla         SLA 超时扫描（定时任务）
└── common      统一返回/异常/AOP日志/限流/幂等
```

### 6.3 核心数据表（概要）
| 表 | 关键字段 |
|---|---|
| `sys_user` | id, username, password, role_id, status |
| `sys_role` | id, role_name, code |
| `sys_permission` | id, perm_code, uri |
| `sys_role_permission` | role_id, permission_id |
| `ticket` | id, ticket_no, title, content, category, priority, status, creator_id, assignee_id, version(乐观锁), sla_deadline, create_time |
| `ticket_log` | id, ticket_id, action, operator_id, remark, create_time（时间线/审计） |
| `ticket_solution` | id, ticket_id, engineer_id, solution_text |
| `evaluation` | id, ticket_id, score, comment |
| `engineer_profile` | user_id, category_skills, current_load |

### 6.4 各技术点如何用到刀刃上（含金量体现）
- **Spring Security + JWT**：RBAC 动态权限（角色-权限-接口映射），JWT 无状态登录；**Redis 存 token 黑名单**实现可控登出与强制下线。
- **Redis**：① 热点工单/字典缓存；② **滑动窗口限流**防刷提交；③ 工单号生成（原子自增）；④ 工程师绩效**排行榜（ZSet）**；⑤ 派单**分布式锁**防并发重复指派；⑥ AI 结果缓存控成本。
- **MyBatis-Plus**：多条件分页、逻辑删除、**乐观锁（version）防并发抢单**、自动填充时间字段。
- **MySQL**：表设计 + **事务**保证「派单 + 状态变更 + 写日志」一致性。
- **AOP**：统一操作审计日志、统一异常处理、接口**幂等**（防重复提交）。
- **定时任务**：SLA 超时扫描标红。
- **AI 工程化**：结构化输出约束、超时/重试、降级兜底、结果缓存——面试讲透这些，比"调了个 API"含金量高。

### 6.5 演示路径（Demo Flow）
报修人提交「我的电脑连不上公司 WiFi」→ AI 返回「类别=网络，优先级=高，建议派给张工，相似工单 3 条」→ 管理员一键采纳派单 → 张工接单、记录解决方案、提交完成 → 报修人 5 星评价 → 管理员看板上「已完成 +1，张工排行上升」。

## 7. 主要风险

| # | 风险 | 影响 | 应对 |
|---|---|---|---|
| R1 | **AI 喧宾夺主**：投入过多精力在 AI，后端反而单薄 | 偏离"后端为主体"定位，简历不够硬 | 严格限定 AI 只占 1 个模块；优先把 RBAC/并发/事务做扎实 |
| R2 | **AI 接口不稳定/有成本**：超时、限流、付费 | 演示翻车、跑不起来 | 同步调用设超时 + 降级人工兜底；结果缓存；demo 前预跑 |
| R3 | **AI 结构化输出不可靠**：返回非法 JSON | 解析失败 | Prompt 强约束 JSON + 校验 + 重试 1 次 + 失败降级 |
| R4 | **范围蔓延**：想加 MQ、微服务、消息推送 | 做不完、超出约束 | 严守 MVP 边界，Out of Scope 项坚决砍 |
| R5 | **并发派单重复指派** | 数据不一致 | 乐观锁 + Redis 分布式锁 + 事务 |
| R6 | **前端拖累进度**：纠结样式 | 时间花错地方 | 用 Element Plus 现成组件，前端"能演示"即可 |
| R7 | **被认为是 CRUD 项目**：面试缺亮点 | 含金量打折 | 主动准备亮点话术：限流/幂等/状态机/RBAC/AI 工程化 |
| R8 | **相似工单检索做太重**：上来就上向量库 | 复杂度爆炸 | MVP 用 类别+关键词 LIKE；向量检索列为可选进阶 |
